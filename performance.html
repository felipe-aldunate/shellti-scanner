Waterfall Estimado de Carga
¿Qué es un Waterfall?
El Waterfall (cascada de carga) muestra de forma visual el orden y duración en que el navegador descarga los recursos de un sitio web — HTML, CSS, JavaScript, imágenes y scripts de terceros — desde el momento en que se hace la primera solicitud hasta que la página está completamente cargada.

Cada barra horizontal representa un recurso. Su longitud indica cuánto tiempo tardó en descargarse. El orden de arriba hacia abajo refleja la secuencia de descarga. Los recursos que bloquean la carga principal (CSS y JS síncronos) son los más críticos.
⚠ Este waterfall es una estimación basada en el dominio analizado, el TTFB real medido, y los recursos típicos de ese tipo de sitio. No es una medición directa del navegador — para eso se recomienda Chrome DevTools o WebPageTest.
Colores de recursos
HTML principal
Documento base de la página. Es el primer recurso que se descarga. Su tiempo equivale al TTFB.
CSS / Estilos
Hojas de estilo. Bloquean el renderizado hasta completar su descarga. Deben estar minificados y en <head>.
JavaScript
Scripts. Pueden bloquear el parser HTML si no usan defer o async. Son el principal culpable de TTI elevado.
Imágenes
No bloquean el renderizado pero afectan el LCP (Largest Contentful Paint). La imagen hero es la más crítica.
Scripts terceros
Analytics, trackers, publicidad. Son los más problemáticos: externos, lentos y fuera de tu control. Cada tracker detectado suma una barra aquí.
Cómo interpretar los tiempos
RangoEvaluaciónQué significa
< 200msExcelenteRecurso muy liviano o bien cacheado. No impacta la experiencia.
200–400msBuenoTiempo normal para recursos externos. Aceptable en la mayoría de contextos.
400–800msRegularEmpieza a ser perceptible para el usuario. Revisar si hay compresión y caché activos.
> 800msCríticoEl usuario percibe la lentitud. Puede causar abandono. Optimización urgente.
Optimizaciones clave
TécnicaImpactoQué hace
defer / async en JSAltoEvita que los scripts bloqueen el parser HTML. El navegador puede seguir procesando el DOM.
Lazy loading imágenesAltoSolo carga las imágenes que están en el viewport visible. Reduce el peso inicial.
CDNAltoSirve los recursos desde un servidor geográficamente cercano al usuario. Reduce latencia.
Compresión Brotli/GzipMedioReduce el tamaño de archivos de texto (HTML, CSS, JS) entre 60–80% antes de transmitirlos.
HTTP/2 MultiplexingMedioPermite descargar múltiples recursos en paralelo sobre una sola conexión TCP.
Preload de recursos críticosMedio<link rel="preload"> le indica al navegador qué descargar primero antes de parsear el HTML completo.
🔍 Para un waterfall real y completo, usa Chrome DevTools (F12 → Network), WebPageTest.org o GTmetrix. Son gratuitos y miden tiempos reales desde la red del usuario.
Score de Performance
¿Qué mide el Score?
El Score de Performance es un puntaje de 0 a 100 que resume qué tan rápido y eficiente es el sitio web para el usuario final. Se calcula combinando múltiples señales técnicas: tiempo de respuesta del servidor, protocolo HTTP usado, compresión, caché, headers de seguridad y presencia de scripts externos que ralentizan la carga.

Es equivalente conceptualmente al score de Google Lighthouse / PageSpeed Insights, aunque este es una estimación basada en indicadores externos observables.
